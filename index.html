Setup & Systemzugang

Willkommen zum Kurs "ABAP Workbench"! Bevor wir mit der Entwicklung beginnen, müssen wir unsere Umgebung einrichten und uns am System anmelden. Diese Schritte basieren auf der Einführungspool (Dozent: Thilo Kunz).

1. Softwareinstallation

Stelle sicher, dass die folgende Software installiert ist:

  • Java SE Development Kit 21.0.8

    Vergiss nicht, die Umgebungsvariablen für den Dateipfad zu setzen (z.B. C:\Program Files\Common Files\Oracle\Java\javapath).

  • SAPGUI for Java 8.10 Rev 1

    Installation via Kommandozeile: java -jar platingui_810_1-80009304.jar.

2. Netzwerkzugang (WLAN)

Die Zugangsdaten für das WLAN im Kursraum lauten:

  • SSID: SAP
  • Passwort: [WLAN-Passwort]

3. SAP-GUI einrichten (System A4H)

Richte die Verbindung im SAP GUI wie folgt ein:

  1. Neue Verbindung hinzufügen.
  2. Zu den "Erweiterten Einstellungen" wechseln.
  3. Den Expertenmodus aktivieren.
  4. Folgenden Connection-String eintragen: conn=/H/[Server-IP]/S/3200.
  5. Als Beschreibung z.B. A4H vergeben.
  6. Speichern.

4. Anmeldung am SAP-System

Verwende diese Daten für die Anmeldung am SAP-System:

  • Client: [Client]
  • User: [User-ID]
  • Password (Initial): [Initial-Passwort] (du wirst aufgefordert, es zu ändern)
  • Logon Language: DE

Nach der Anmeldung landest du im SAP Easy Access Menü. Der "erste Schritt" in die Workbench erfolgt über die Transaktion SE80.

ABAP Grundlagen & Architektur

ABAP (Advanced Business Application Programming) ist die Kernprogrammiersprache für die meisten Geschäftsanwendungen im SAP-System, insbesondere SAP S/4HANA und SAP ERP.

SAP-Systeme nutzen eine 3-Schichten-Architektur:

  1. Präsentationsschicht (UI): Die Oberfläche, mit der Benutzer arbeiten, z.B. SAP GUI oder Fiori im Browser.
  2. Anwendungsschicht (Applikation): Hier läuft dein ABAP-Code. Der ABAP Application Server verarbeitet die Geschäftslogik und Berechtigungen.
  3. Datenbankschicht (Datenbank): Speichert die Daten (z.B. SAP HANA).
ABAP-Entwickler arbeiten primär in der Anwendungsschicht und nutzen das ABAP Dictionary sowie Open SQL für datenbankunabhängige Zugriffe.

Zentrale Artefakte sind Programme (Reports), Klassen, Funktionsbausteine, die Paketstruktur und Transporte zur Verteilung von Änderungen zwischen Systemen (DEV → QAS → PRD).

Das SAP-Flugmodell (Kontext)

Als zentrale Schulungs- und Beispiel-Datenbank verwendet SAP (und auch das BC400-Skript) das sogenannte Flugmodell. Es beschreibt die Beziehungen zwischen Fluggesellschaften, Flugverbindungen, Flügen und Buchungen.

Die Kenntnis der wichtigsten Tabellen ist essenziell, um die meisten ABAP-Beispiele zu verstehen:

  • SCARR: Die Stammdatentabelle für Fluggesellschaften (z.B. 'LH' für Lufthansa, 'AA' für American Airlines). Enthält den Namen der Gesellschaft und die Währung.
  • SPFLI: Die Tabelle für Flugverbindungen (der "Flugplan"). Sie definiert, welche Gesellschaft (CARRID) regelmäßig eine bestimmte Strecke (CONNID) fliegt, z.B. von AIRPFROM (Abflughafen) nach AIRPTO (Zielhafen).
  • SFLIGHT: Die Tabelle für konkrete Flüge. Sie bezieht sich auf eine Flugverbindung (CARRID, CONNID) und fügt ein Datum (FLDATE) hinzu. Hier sind auch Daten zur Belegung (SEATSMAX, SEATSOCC) gespeichert.
  • SBOOK: Die Tabelle der Flugbuchungen. Jede Buchung bezieht sich auf einen konkreten Flug (CARRID, CONNID, FLDATE) und einen Kunden (CUSTOMID).

Diese Tabellen werden im ABAP Dictionary (SE11) definiert und in fast allen Beispielen zu Open SQL, ALV, Dynpros und Web Dynpros verwendet.

Entwicklungsumgebung (SE80/ADT)

Entwicklung erfolgt klassisch in der SE80 (Object Navigator) oder modern in ABAP Development Tools (ADT) in Eclipse. Der empfohlene Weg ist ADT für moderne Syntax, Quick Fixes, Unit Tests und integrierte Tools.

Artefakte werden in Paketen organisiert. Lokale Objekte ($TMP) sind transportfrei, produktive Objekte benötigen ein Paket und eine Transportaufgabe.

Datentypen & Datenwörterbuch

ABAP kennt elementare Typen (z.B. I, P, STRING, D, T) sowie abgeleitete Typen aus dem ABAP Dictionary.

Das Dictionary (Transaktion SE11) definiert Domänen (technische Typen), Datenelemente (semantische Bedeutung/Texte) und Tabellentypen. Es stellt Konsistenz zwischen Programmen und Datenbank sicher (Längen, Wertebereiche, F4-Hilfen, Texte).

Interne Tabellen

Interne Tabellen sind das zentrale Arbeitsmittel zur Verarbeitung von Datenmengen im Speicher. Typen: Standard (ohne eindeutigen Schlüssel), Sortiert (sortierter Schlüssel), Hash (eindeutiger Hash-Schlüssel).

Häufige Operationen: APPEND, INSERT, READ TABLE ... WITH KEY ..., LOOP AT, SORT, DELETE. In moderner Syntax kommen Ausdrücke wie FILTER, REDUCE und FOR hinzu.

Wichtiger Hinweis (Kopfzeilen): In den BC400-Skripten wird der Unterschied zwischen modernen internen Tabellen (mit explizitem Arbeitsbereich, z.B. DATA gs_data LIKE LINE OF gt_data) und der veralteten Syntax mit Kopfzeile (Header Line) betont. Bei der Kopfzeilen-Syntax (z.B. DATA itab TYPE table WITH HEADER LINE) dient die Tabelle selbst implizit als Arbeitsbereich. Diese alte Syntax sollte in neuen Entwicklungen (wie in allen Beispielen auf dieser Seite) nicht mehr verwendet werden, da sie zu unleserlichem Code führt.

Modularisierung in ABAP

Wiederverwendung und Strukturierung über Unterprogramme (FORM), Funktionsbausteine (Transaktion SE37) (remote-fähig via RFC), Methoden in Klassen (Transaktion SE24) und Includes zur Code-Aufteilung.

Eine spezielle, standardisierte Form von Funktionsbausteinen sind BAPIs (Business Application Programming Interfaces). Sie werden im Business Object Repository (BOR) (Kap. 4, Lektion 5) verwaltet und dienen als stabile Schnittstellen, um auf SAP-Geschäftsobjekte (z.B. einen Kundenauftrag) zuzugreifen. Sie sind immer RFC-fähig und werden oft von externen Systemen genutzt.

Best Practice: Neue Entwicklungen objektorientiert (globale/lokale Klassen), klare Schnittstellen und Unit Tests.

Open SQL Grundlagen

Open SQL kapselt DB-Zugriffe datenbankunabhängig. Kernbefehle: SELECT, INSERT, UPDATE, DELETE, MODIFY. Moderne Syntax nutzt Inline-Deklaration (@DATA) und INTO @DATA(itab).

Joins, Aggregationen, WHERE, ORDER BY, GROUP BY sind verfügbar. Für semantische Modellierung und HANA-Nutzung (Stichwort "Code Pushdown"): ABAP CDS Views mit Annotationen (z.B. für OData/Analytik).

Selektionsbild & klassische Listen

Reports (Transaktion SE38) definieren Eingaben via PARAMETERS und SELECT-OPTIONS (SELECTION-SCREEN). Klassische Ausgabe per WRITE, einfache Formatierung, Hotspots und basic Interaktion.

ALV (Simple/OO)

Für tabellarische Ausgaben: ALV. Modern bevorzugt: CL_SALV_TABLE für schnelle OO-ALV-Listen. Alternativ klassische Funktionsbausteine (REUSE_ALV_GRID_DISPLAY).

Hinweis (Kap. 9): Eine weitere wichtige Variante ist die Klasse CL_GUI_ALV_GRID. Im Gegensatz zu CL_SALV_TABLE (das den Bildschirm "übernimmt"), wird CL_GUI_ALV_GRID verwendet, um ein ALV-Gitter in einen Container auf einem klassischen Dynpro (siehe Kapitel 8 des Skripts) einzubetten.

Funktionen: Sortieren, Filtern, Summen, Layouts sichern, Export.

Debugging & Fehlerbehandlung

Fehlerbehandlung per Ausnahmen (klassisch RAISE, OO RAISE EXCEPTION), MESSAGE-Klassen (Transaktion SE91), TRY...CATCH. Debugger (ADT/SE80) unterstützt Breakpoints, Watchpoints, Variablen-Inspektion. Bei Laufzeitfehlern hilft die Transaktion ST22 (Dump-Analyse).

Qualitätssicherung über ABAP Unit und ATC (Code-Checks). (Ergänzung zu Kap. 11): Die technische Basis für ATC ist oft der Code Inspector (Transaktion SCI), der detaillierte statische Analysen (Performance, Sicherheit, Syntax) für ABAP-Objekte durchführt.

Entwicklungsumgebung (SE80/ADT)

Entwicklung erfolgt klassisch in der SE80 (Object Navigator) oder modern in ABAP Development Tools (ADT) in Eclipse. (Kap. 12) ADT ist eine Eclipse-basierte IDE [cite: 4989], die sich über ein "ABAP-Projekt" [cite: 5040, 5052] mit dem SAP-System verbindet. Sie bietet native Editoren für Programme und Klassen [cite: 5098, 5104] und integriert SAP-GUI-Fenster für ältere Werkzeuge (wie Dynpro oder Dictionary-Objekte), für die es keinen nativen Editor gibt[cite: 5097, 5126].

Artefakte werden in Paketen organisiert. Lokale Objekte ($TMP) sind transportfrei, produktive Objekte benötigen ein Paket und eine Transportaufgabe.
